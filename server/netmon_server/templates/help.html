How it works

What each probe actually measures, how often, and how to read the numbers. Intervals and targets below are the defaults — every monitor can override them in its monitor.ini.

🏓 Latency & packet loss (ping)

Every 2 s the monitor sends a single ICMP ping (ping -c 1 -W 2) to three targets:

targetaddresswhat it tells you
gatewayauto-detected from the default route health of the local link — any loss here means cable, Wi-Fi, switch or router
quad99.9.9.9 the internet via two independent anycast networks — one of them having a bad day doesn't look like your outage
google8.8.8.8

The reported latency is the round-trip time (RTT) of each reply. A ping with no reply within the timeout counts as LOSS. Loss percentage is simply lost / sent over the selected period or chart bucket. Thresholds used across the UI: ≤ 0.1 % OK > 0.1 % minor loss > 1 % problem

🛑 Outage detection

Outages are derived from the ping rounds (all three targets are pinged at the same moment, so they can be judged together):

  • local — the gateway didn't answer. The problem is on your side of the wall socket, whatever the public targets did.
  • internet — the gateway answered but both public targets were lost in the same round. That's the provider, not your LAN.
  • Losing a single public target creates no event — one unlucky packet or one anycast hiccup is noise, not an outage.

Consecutive bad rounds of the same kind merge into one event; a gap longer than 2.5× the ping interval splits it (so time when the monitor itself wasn't running never bridges two outages). The minimum reported duration is one ping interval.

🌐 Service reachability — DNS / TCP / TLS

Every 30 s the monitor opens https://www.google.com/generate_204 from scratch and times each phase separately:

phasemeasured asslow / failing usually means
DNSresolving the hostname (getaddrinfo) resolver problems — your router's DNS, the ISP resolver, or DNS interception
TCPTCP connect to port 443 path congestion or packet filtering; roughly one network round trip
TLSTLS handshake on the open connection adds ~1–2 round trips + crypto; jumps when the path degrades or middleboxes interfere

Then an HTTP GET must return a status code (expected 204). If any phase fails or times out (10 s total budget), the sample is recorded as FAIL. This is the probe that catches the classic “ping works but the internet doesn't” — ICMP can flow while DNS is down or TCP/443 is being dropped. Sustained FAILs also trigger an e-mail alert even when ping loss is zero.

⬇️ Download speed

Every hour the monitor downloads a test file from speed.cloudflare.com (default 50 MB) over a single HTTPS connection and reports bytes × 8 / seconds in Mbit/s — including DNS, connection setup and TLS, like a real download.

Adaptive sizing: if the file finishes in under 3 s, the line is fast enough that TCP ramp-up dominates and the result underestimates reality. The monitor then immediately re-measures once with a payload sized for roughly a 6 s download (capped at 200 MB) and stores that result instead.

Things to keep in mind when reading the chart: it's one TCP stream (multi-stream tests like Ookla can read higher), and it shares the line with whatever the household is doing — a dip during the evening Netflix hour may be your own traffic, not the ISP. The hourly cadence exists to show patterns (peak vs. night), not to benchmark.

🌍 Public IP

Every 15 min the monitor asks a plain-text echo service (api.ipify.org) what IP it appears from, and records it only when it changes — an ISP handover, a reconnect after an outage, or a CGNAT move. The network page shows the current IP with its reverse-DNS name (which usually reveals the ISP), and each change appears as a teal 🌍 marker in the charts, so it's easy to see whether an outage coincided with getting a new address.

💓 Heartbeat & measurement coverage

Every 60 s the monitor writes a heartbeat; on a clean shutdown it writes a final STOP. A gap longer than 150 s between heartbeats means the monitor wasn't measuring at all. If the last event before the gap was a STOP, the gap is shown as a controlled stop; otherwise as a crash / powered-off host. Coverage is the share of the period the monitor was actually running — outages can only be detected while it measures.

Monitor unreachable ≠ network outage: monitors measure and store everything locally; the server just pulls their data every 30 s. When a monitor can't be reached, the dashboard flags it, and once the connection returns, the buffered measurements sync over — the history fills in retroactively.

📊 Reading the charts

  • Bucketing: chart series are averaged into time buckets (min 60 s, sized so a chart has at most ~1500 points). Loss per bucket is the loss percentage within that bucket, which is why a single lost ping shows as a small bump rather than 100 %.
  • Shaded bands are detected outages: red = local link amber = internet / ISP
  • Yellow dashed lines are your notes (📝) — hover for the text, click a chart to add one at that moment.
  • Calendar heatmap: one cell per day, colored by that day's packet loss on the public targets only (the gateway is excluded — it would mix LAN problems into an internet-quality view). Same thresholds as everywhere else; gray = the monitor wasn't measuring that day.
  • Speed chart shows raw hourly points, not bucketed averages.
